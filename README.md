# Аналіз якості тендерної документації Prozorro

Це робочий MVP для пошуку потенційних ризиків і нечітких вимог у публічних тендерах Prozorro. Система завантажує живі дані з Prozorro API, кешує метадані та документи, витягує текст із практичних форматів, рахує бал якості 0-100 і показує результати в українській панелі.

Система не робить юридичних висновків. Усі сигнали подані як потенційний ризик, можлива проблема або нечітка вимога і потребують перевірки людиною.

## Що входить

- обробка одного тендера за Prozorro UUID;
- обробка партії з 5 нових придатних тендерів;
- фільтр очікуваної вартості 1 000 000-40 000 000 грн;
- завантаження і кешування документів;
- парсинг PDF, DOCX, TXT, HTML, XLSX, XLS, CSV і, де доступно, старого DOC через `textutil` або `pandoc`;
- детерміновані правила для брендів, відсутності «або еквівалент», листів виробника, географічних обмежень, нечітких формулювань, надмірних кваліфікаційних і документальних вимог, умов оплати/поставки, проєкту договору та складних для аналізу документів;
- SQLite-сховище результатів;
- українська панель зі списком тендерів, деталями, доказовими цитатами, поясненнями і запропонованими переписуваннями;
- опційне коротке збагачення пояснень через локальний Codex CLI без повнодокументного LLM-аналізу.

## Встановлення

```bash
python3 -m venv .venv
source .venv/bin/activate
pip install -r requirements.txt
```

## Обробка живих тендерів

Партія з 5 нових тендерів:

```bash
python -m prozorro_quality.cli batch --limit 5 --max-pages 12
```

Один тендер за UUID з Prozorro API:

```bash
python -m prozorro_quality.cli process 73e491ce38214a5da0589d0564408609
```

З коротким Codex-збагаченням пояснень:

```bash
python -m prozorro_quality.cli batch --limit 5 --max-pages 12 --use-codex
```

Codex отримує лише вже знайдені короткі сигнали й цитати, а не повні документи. Якщо Codex CLI недоступний або завершується з помилкою, система зберігає детерміновані пояснення і додає обмеження до результату.

## Запуск панелі

```bash
python -m prozorro_quality.cli serve --host 127.0.0.1 --port 8000
```

Відкрийте `http://127.0.0.1:8000`. На головній сторінці можна запустити обробку нової партії, відфільтрувати результати, відкрити тендер і переглянути:

- загальний бал;
- підбали: повнота, зрозумілість, конкурентність, технічна нейтральність, якість проєкту договору;
- опрацьовані документи та статус парсингу;
- сигнали за категоріями;
- доказові цитати;
- коротке пояснення українською;
- запропоноване переписування;
- обмеження автоматичного аналізу.

## React dashboard

Frontend живе окремо в `frontend/` як npm/Vite/React/TypeScript застосунок.

```bash
cd frontend
npm install
npm run dev
```

Для збірки bundle, який підхоплює FastAPI-шаблон:

```bash
cd frontend
npm run build
```

Збірка записує артефакти в `prozorro_quality/static/dashboard/`, а Python-застосунок завантажує `dashboard/dashboard.js`.

## Налаштування

Можна змінити параметри через змінні середовища:

```bash
PROZORRO_MIN_VALUE=500000
PROZORRO_MAX_VALUE=40000000
PROZORRO_BATCH_SIZE=5
PROZORRO_MAX_PAGES=12
PROZORRO_QUALITY_DATA_DIR=./data
PROZORRO_REQUEST_TIMEOUT=8
PROZORRO_CODEX_COST_MODEL=gpt-5.5
PROZORRO_CODEX_INPUT_USD_PER_M=5.0
PROZORRO_CODEX_CACHED_INPUT_USD_PER_M=0.5
PROZORRO_CODEX_OUTPUT_USD_PER_M=30.0
```

Дані зберігаються у `data/quality.sqlite3`, кеш метаданих і документів у `data/cache/`.

Система бере точний token usage з `codex exec --json` для коротких Codex-збагачень і рахує орієнтовну API-вартість за ставками `PROZORRO_CODEX_*`. За замовчуванням у цьому середовищі використано модель вартості `gpt-5.5`; старі записи без token usage показуються як `не рахувалось`.

## Тести

```bash
pytest
```

Тести покривають базові правила виявлення, обережність формулювань, scoring, вибір тендерів і SQLite-зберігання.

## Ітерації на реальних даних

Після початкового MVP виконано 11 повних партій по 5 реальних тендерів Prozorro. Загалом збережено 55 опрацьованих прикладів із різних секторів. Критичний перегляд цих результатів привів до зменшення false positive для «або аналог», Microsoft Excel як формату файлу, списків технічної сумісності, контактної адреси учасника та стандартного ARMA/нотаріального контексту.

Деталі: `iteration_report.md`.

## Відомі обмеження MVP

- правила не замінюють експертний аналіз закупівельної документації;
- скановані PDF і підписані файли можуть не давати достатнього тексту;
- старий DOC залежить від наявності `textutil` або `pandoc`;
- пошук останніх придатних тендерів потребує кількох сторінок API, бо багато свіжих записів мають малу вартість або не містять придатних документів;
- бал є практичним індикатором якості документації, а не юридичною оцінкою.
