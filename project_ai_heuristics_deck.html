Технічна теза

AI ефективний, коли покращує детерміністичну систему, а не замінює її

Цей проєкт аналізує тендерні документи через Python rule engine. AI використовується для discovery, критики, генерації патчів і audit-у, а production behavior лишається відтворюваним, тестованим і дешевим.

Проблема

Ризикові умови заховані не в одному полі, а в сотнях сторінок документів

Тендерна якість не зводиться до одного класифікатора. Обмеження конкуренції можуть бути в технічних специфікаціях, проєкті договору, xlsx-опитувальниках, додатках до кошторису, вимогах до персоналу або умовах логістики.

Вхідні дані

DOC/DOCX/PDF/XLS/XLSX із різною якістю тексту, сканами, таблицями, змішаною мовою і юридичними шаблонами.

Потрібний вихід

Не “вирок”, а пояснюваний risk signal: категорія, severity, evidence quote, пояснення і можливе переписування.

Головна складність

False positives коштують довіри. Система має відрізняти “АВК як вимога до пропозиції” від “АВК у післяконтрактних актах”.

Чому AI сам по собі не ідеальний

LLM добре читає документи, але як production oracle він дорогий, нестабільний і складний для regression testing.

Архітектура

Runtime шлях лишається простим: текст → правила → issue → score → UI

AI не стоїть у критичному production path. Основна система працює детерміністично і може масово переобробляти тендери без повторних LLM-витрат.

AI development loop

AI прискорює discovery, але результатом стає diff у коді

Це головна причина ефективності: AI читає різноманітні документи й пропонує евристики, але кожна евристика стає Python-кодом, тестом і audit artifact-ом.

Система росла через короткі цикли: документи → критика → правило → audit

Чому це працює

AI зменшує вартість пошуку правил, а deterministic code зменшує вартість експлуатації

Це не “AI замість програми”. Це AI як інструмент для швидкого нарощування програмної системи, де кожна зміна має diff, тести і передбачуваний runtime.

Rule engine

Кожна евристика є структурованим Python object, не прихованим prompt-ом

Поведінка системи локалізована: `Rule` описує що шукати і як штрафувати, `should_skip_match()` відсікає safe contexts, `score_issues()` переводить issues у бали.

Shape правила

@dataclass(frozen=True)
class Rule:
    category: str
    title: str
    severity: str
    pattern: re.Pattern[str]
    explanation: str
    suggested_rewrite: str
    subscores: tuple[str, ...]

Rule одночасно є detector-ом, explainability unit і scoring metadata.

Execution loop

for parsed in parsed_documents:
    for rule in RULES:
        for match in rule.pattern.finditer(parsed.text):
            quote = evidence_window(text, match.start(), match.end())
            if should_skip_match(rule.category, quote):
                continue
            issues.append(Issue(...))
            break

Один match на rule/document, evidence window до 700 символів, потім dedupe + score.

Базові правила ловлять конкуренційні обмеження, які повторюються в різних секторах

Ці правила не намагаються довести порушення. Вони підсвічують текстові конструкції, які часто потребують review.

Validation

Quality gate: евристика не потрапляє в систему без audit trail

AI корисний не лише для генерації, а й для критики. Але фінальний gate має бути повторюваним: tests, counts, audit JSON, і бажано людська юридична валідація.

Наступний природний крок: юридичний feedback loop. Юристи можуть підтверджувати, пеналізувати або знижувати confidence евристик; система перетворює ці рішення в labels, tests і нові guard-и.

Adversarial adaptation

Якщо замовники почнуть обходити правила, система може навчатися на самих обходах

Rule engine не є статичним списком. Він може вести історію формулювань, buyer-level telemetry і швидко додавати нові правила після появи нових шаблонів обходу.

Напрямки розвитку

Проєкт може вирости з detector-а в систему procurement intelligence

Ключова перевага вже закладена: кожне правило має evidence, версію, audit і можливість бути підтвердженим або оскарженим експертом.

Юридична валідація

Lawyer labels: true risk, weak signal, false positive, legal exception. Labels стають regression tests і rule weights.

Buyer risk profiles

Історія замовників: повторювані патерни, обходи, частота підтверджених сигналів, зміна поведінки після підсвітки.

Adaptive rule mining

Періодичний AI mining нових документів і user-reported misses; автоматичний список кандидатів на review.

Confidence model

Не всі правила однаково сильні. Додати confidence, source type, legal validation status і “needs review”.

Parser hardening

Краща робота зі сканами, таблицями, XLSX, старими DOC, OCR і документами зі слабким текстовим шаром.

Explainable API

API, де кожен issue має rule id, regex version, evidence, skip decision і посилання на повний документ.

Takeaway: model discovers, code decides. AI дає leverage на читання документів і пошук нових формулювань; детерміністичний Python дає контроль, швидкість, тестованість і довіру.
